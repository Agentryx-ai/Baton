# 휴지통 이동·복원 뒤 session URL과 `새 대화` 본문이 불일치함

## 상태

- 상태: 미해결
- 발견일: 2026-07-22
- 우선순위: P2

## 증상

현재 선택한 프로젝트리스 대화를 휴지통으로 이동하면 본문은 `새 대화`로 초기화되지만
주소는 이동한 대화의 `?session=<id>`를 계속 유지한다. 휴지통에서 같은 대화를 복원하고
`대화로 돌아가기`를 눌러도 활성 목록에는 복원된 대화가 보이는 반면, 본문과 선택 표시는
계속 `새 대화`로 남는다.

주소가 가리키는 canonical session과 화면의 선택 상태가 달라 사용자는 복원이 실패했거나
빈 대화로 이동했다고 오인할 수 있다. 목록에서 해당 대화를 다시 클릭해야 URL, 선택, 본문이
비로소 일치한다.

## 재현

검수 session: `239e1c3e-c024-4da0-9759-f52f4e915b8c`

1. 프로젝트리스 `새 대화`에서 짧은 turn을 완료한다.
2. 선택된 대화를 `휴지통으로 이동`하고 확인한다.
3. 본문과 URL을 확인한다.
4. 휴지통에서 같은 대화를 복원한다.
5. `대화로 돌아가기`를 누르고 활성 목록, 본문, URL을 비교한다.

실측 결과:

```text
archive/restore API: 성공
휴지통 자동삭제일과 복원 버튼: 정상 표시
복원 뒤 활성 목록: 해당 대화 표시
URL: ?session=239e1c3e-c024-4da0-9759-f52f4e915b8c
본문/선택 상태: 새 대화
목록 항목을 다시 클릭한 뒤: 해당 session 본문으로 정상 전환
```

QA session은 검수 후 공식 archive API로 다시 휴지통에 이동했으며 영구 삭제하지 않았다.

## 원인

- `archivePendingSession()`은 archive와 목록 새로고침만 수행하고, 현재 선택된 session을
  archive한 경우 URL과 선택 상태를 함께 정리하거나 명시적인 다음 화면으로 이동하지 않는다.
- `restoreSession()`도 restore와 목록 새로고침만 수행하며, 복원한 session을 선택하거나 기존
  `?session=` query와 본문 상태를 다시 동기화하지 않는다.
- `대화로 돌아가기`는 scope만 active로 바꾸므로 불일치가 그대로 노출된다.

## 영향

- deep link와 화면 본문이 서로 다른 대화를 나타낸다.
- 새 메시지를 어느 session에 보내게 될지 사용자가 신뢰하기 어렵다.
- 휴지통 이동·복원이 성공했어도 실패처럼 보이며 추가 클릭이 필요하다.

## 완료 조건

- 현재 선택된 session을 archive하면 URL, 선택 상태, 본문을 하나의 명시적 상태로 원자적으로
  전환한다.
- 대화를 restore하면 복원된 session을 선택하거나, URL에서 session query를 제거해 본문과
  주소가 항상 일치하도록 한다.
- `대화로 돌아가기`, browser back/forward, reload 뒤에도 URL·목록 선택·본문이 동일한
  session을 나타낸다.
- 프로젝트리스 대화의 create → archive → restore → archive E2E에서 상태 불일치를 검증한다.
